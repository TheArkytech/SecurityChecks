[THREAT-2026-0035] "Comment and Control" — Prompt injection via GitHub PR/issue
    metadata → AI agent credential theft (Claude Code, Gemini CLI, Copilot)
- Category: AI Dev / Prompt Injection
- Impact: 🔴 Directly affects us (we use Claude Code; any workflow using AI agents
  on GitHub events is vulnerable)
- Summary: Disclosed ~April 15 by Aonan Guan et al. (Wyze Labs / Johns Hopkins).
  Any GitHub user who opens a PR or files an issue with a crafted title/body can
  trick Anthropic's Claude Code Security Review, Google's Gemini CLI Action, or
  GitHub Copilot Agent into exfiltrating CI/CD secrets (API keys, GitHub tokens)
  through PR comments or git commits — no victim interaction required. Root cause:
  untrusted GitHub metadata flows directly into AI agents that hold production
  secrets and unrestricted tool access in the same runtime. No CVEs assigned.
  Anthropic rated CVSS 9.4 internally, paid $100 bounty, then re-classified
  severity to "None." Google paid $1,337. GitHub paid $500, called it an
  "architectural limitation." No public advisories or patches from any vendor.
- Recommended action: Audit every GitHub Actions workflow using an AI coding agent
  — if triggered by pull_request, issues, or issue_comment and has access to secrets,
  restrict secret access to protected-branch workflows only; never pass raw PR/issue
  metadata (title, body, comments) into an AI agent that holds credentials
- Source: https://www.securityweek.com/claude-code-gemini-cli-github-copilot-agents-vulnerable-to-prompt-injection-via-github-comments/

[THREAT-2026-0036] CVE-2026-42945 "NGINX Rift" — 18-year-old unauthenticated heap
    buffer overflow in rewrite module (CVSS 9.2, actively exploited)
- Category: CVE / Infrastructure / Web Server
- Impact: 🟠 Could affect us (nginx is ubiquitous in web infra and CI/CD containers)
- Summary: Heap buffer overflow in ngx_http_rewrite_module present in every nginx
  build since 2008. Disclosed May 13 by F5 + depthfirst. Single crafted HTTP request
  triggers the overflow when nginx processes rewrite/if/set directives with unnamed
  capture groups ($1, $2). Reliable DoS; RCE possible on some configurations.
  Public PoC on GitHub. Active exploitation confirmed by VulnCheck on May 16 — only
  3 days after disclosure. Affects NGINX OSS 0.6.27–1.30.0, Plus R32–R36, and
  Ingress NGINX ≤1.15.1. CISA KEV inclusion expected.
- Recommended action: Run `nginx -v` on all servers/containers; upgrade to 1.30.1
  (stable) or 1.31.0 (mainline) and restart workers; for NGINX Plus apply F5 advisory
  K000161019 (R32 P6 / R35 P2 / R36 P4); for Ingress NGINX update the controller
- Source: https://thehackernews.com/2026/05/18-year-old-nginx-rewrite-module-flaw.html
          https://www.helpnetsecurity.com/2026/05/18/ngnix-vulnerability-exploited-cve-2026-42945/
          https://www.akamai.com/blog/security-research/nginx-critical-heap-buffer-overflow-cve-2026-42945

TIP OF THE DAY

The GitHub tag→fork exploit (used in Laravel-Lang and actions-cool attacks) is
a platform trust issue: GitHub allows tags to silently point to commits in any fork,
not just the original repo. Defense is simple — audit your workflows and configs for
any mutable tag references:

  grep -r 'uses:' .github/workflows/ | grep -v '@[0-9a-f]\{40\}'

Any result without a 40-char SHA is a potential tag→fork attack surface. For
Composer/npm, verify package versions against published SHAs where possible.
StepSecurity's harden-runner also detects tag-redirect attacks at runtime.
